## 1. インデックスページの分岐軸

選んでからお手前に入る。軸は以下：

1. **流儀**：表千家 / 裏千家 / （両方並記）
2. **季節 = 炉 / 風炉**　※「夏点前=風炉(5〜10月)」「冬点前=炉(11〜4月)」は**同じ軸**。重複させない。
3. **性別**：男点前 / 女点前（柄杓・蓋置の扱い、釜蓋に袱紗 等が違う）

> **最初に作るのは確定**：**表千家 × 風炉（夏点前）× ＿＿（男 or 女、要確認）** の薄茶 運び点前。
> 1本作って型を固め、他の枝はそこからフォークして増やす。

データ構造イメージ：
```
表千家 / 風炉(夏) / 女点前 / 薄茶 / 運び  → steps[ {手1}, {手2}, ... ]
        ↑ ここから1本目を作る
```

## 2. コア設計：描画とデータを分離する（最重要）

イラストか Three.js かで割り切れない問題は、**「手順データ層」を描画から切り離す**ことで解消する。

```
[手順データ層]  ← 流儀/季節/性別/一手ごとの所作・道具状態・ポイント（描画非依存）
      │
      ├─→ [イラスト描画]   ← 今すぐ作る。自分目線の絵＋道具アニメ
      └─→ [Three.js描画]   ← 後で差し替え。同じデータでGLBモデルを動かす
                              ↑ Fable5に渡す土台
```

- 同じ23手データを、**当面はイラストで描き、将来 Three.js ビューに差し替え可能**にする。
- これにより「イラストで作り始める」と「Three.jsの土台を用意する」が**両立**する。二択にしない。

### 一手（step）のデータ項目（案）
- `n` 手番 / `title` / `text`（なぜ・どこに・どう）/ `hint`（所作のかなめ）
- `tools`：各道具の状態（位置・蓋の開閉・持ち手・動作中フラグ）← イラストもThree.jsもこれを読む
- `gender_diff` / `season_diff`：男女・炉風炉の差分だけ上書き

## 4. Three.js / AIで「ほぼ無料の3D」調査結果（2026-06）

ユーザー仮説「今のAIなら釜・茶碗の3Dをほぼ無料で作れる」は **概ね正しい**。

| 手段 | 概要 | コスト | Three.js適性 |
|------|------|--------|--------------|
| **Hunyuan3D 3.1** | 画像/テキスト→3D。**GLB(glTF)・USDZ 出力**、Three.js完全対応。オープンソースで自己ホスト可 | 自己ホストなら実質無料 | ◎ そのまま読込 |
| **TRELLIS (Microsoft)** | 1枚画像→PBR付きGLBメッシュ。production級。OSS | 無料(自己ホスト/HF) | ◎ |
| Tripo AI / Meshy | 商用。テキスト→GLB、リトポ・リグまで高速 | 無料枠あり/有料 | ◎ |

**推奨パイプライン（Fable5に渡す土台）**：
1. オリジナル茶碗の**写真 or 自作イラスト**を用意 →
2. **TRELLIS / Hunyuan3D で画像→GLB** に変換 →
3. Three.js に読み込み、最低限のシーン（畳・点前座・ライティング）を組む →
4. この **GLB資産＋シーン雛形を Fable5 に渡し**、リアルなシミュレーションへ昇華。

→ つまり「画像1枚からGLB」が今や安価に回るので、**オリジナリティのある茶碗を画像で作る → GLB化**が現実的な最短路。
